## Intent (why this round of work exists)

Not a task list — the reasoning Fable's plan should trace back to. Pull the
concrete backlog from `HANDOFF.md`'s "What remains" section and the PRD's
§11 open items; both are live documents, re-read them, don't take this
brief's summary as frozen truth.

1. **The MVP is feature-complete and QA'd, but "post-MVP polish and v2
   scoping" is still open.** `HANDOFF.md` names three remaining threads:
   deploy readiness (repo is GitHub-Pages-ready as-is — confirm, don't just
   assume), and two PRD §11 v2 candidates (bridge-to-real-Onshape
   micro-tasks, a fuller commit-a-guess predict interaction if the current
   light tap proves too breezy) plus a stretch exercise (E5 — sketch-level
   vs. part-level features, PRD §7).
2. **The unit-toggle work that just shipped exposed a real authoring
   footgun**, documented in `HANDOFF.md`: exercise-copy fields that embed a
   converted length must be object-literal *getters*, not plain strings, or
   they silently freeze at whatever unit was active on page load. A
   code-review pass caught five instances of this already slipping through
   once. Worth deciding whether that's an acceptable ongoing cost (case-by-
   case discipline, documented) or whether it's worth a more systematic fix
   before E5 (or any future exercise) adds more unit-bearing copy. This is a
   genuine architecture judgment call — exactly the kind of thing Fable
   should own rather than delegate.
3. **Minor hygiene noticed but not chased down this session:** a missing
   favicon causes a 404 in the console on every load (cosmetic, but "zero
   console errors" is this project's own QA bar per `AGENTS.md`); a
   `localStorage` read/write wrapper now exists in both `svg.js` (unit
   preference) and `app.js` (`cad-gym.v1` progress) with slightly different
   shapes — low-severity duplication, Fable's call whether it's worth
   unifying or leaving as two small, independent, easy-to-read helpers.
4. **Human final review is still outstanding** (`HANDOFF.md`) — Justin
   playing E1–E4 end-to-end before anything is deployed anywhere public.
   That checkpoint is Justin's, not Fable's — don't treat an agent QA pass as
   a substitute for it, but do make sure the app is in a state worth him
   reviewing (no known regressions, HANDOFF.md accurate) before flagging it
   ready.

## Constraints (hard rules — do not relitigate)

From `AGENTS.md` (project) and `PRD.md`:

- Vanilla HTML/CSS/JS, static, no backend, no build step, no accounts —
  unless Justin explicitly asks for a rebuild.
- No real CAD engine / constraint solver. Consequences stay pre-authored,
  hand-choreographed scenes, not computed.
- Every exercise (present and future) ships with a full loop — intent,
  predict, choose, change request, consequence, takeaway, **counter-context
  required, no exceptions.**
- Copy is plain-language-first, CAD-term-second.
- Progress lives in browser `localStorage` only.
- Before declaring anything done: run the app in a real browser at desktop
  and mobile widths, complete the exercises touched, confirm zero console
  errors, and **update `HANDOFF.md`** with what changed and what remains.
  (This brief's own predecessor commit skipped that step once this session —
  don't repeat it.)

## Suggested workflow

1. Re-read `HANDOFF.md`, `PRD.md` §11, and this brief. Confirm the intent
   above is still accurate (these are live docs; something may have moved).
2. Write the plan: a short ordered list of concrete tasks, each with a named
   executor (`codex`, `opus`, `sonnet`, or "Fable directly"). Post it before
   executing — even without a hard approval gate, thinking out loud here is
   cheap insurance against building the wrong thing.
3. Dispatch each task via `Agent` (spawn-and-wait, never poll). Mechanical /
   read-only-sweep work → `codex`. Judgment-flavored execution → `opus` or
   `sonnet`. Keep architecture calls (item 2 above), visual/copy taste, and
   final integration for yourself.
4. **Evaluate every result before committing it.** Check against: the intent
   section above, the hard rules, and — specifically, since it just bit this
   project once — whether any new/changed exercise-copy field that embeds a
   converted length is a getter, not a plain string. Run the app; don't
   review by reading the diff alone.
5. Commit in reviewed units (not one giant commit at the end), with commit
   messages that say why, not just what.
6. Update `HANDOFF.md` before calling anything done.

## Evidence required before calling a task done

- Actually run the app (desktop + mobile width) and exercise whatever
  changed — screenshot or described observation, not "the code looks right."
- Zero console/page errors.
- No exercise (existing or new) ships without a working counter-context.
- `HANDOFF.md` reflects what changed and what's still open.
- If Fable delegated a mechanical slice: exactly one `Agent → codex` spawn
  per task, zero `codex exec`/`pgrep`/`lsof`/`ps` calls from Fable's own
  turns.
